Bug: Peer sent MsgUnchoke (ID=1) instead of MsgBitfield (ID=5) as first message.

Some peers skip the bitfield message entirely if they have all pieces and go
straight to unchoke. The fix was to check the first message ID and if it's
an unchoke, assume the peer has all pieces and create a full bitfield with
all bits set to 1 (0xff per byte).

Lesson: BitTorrent peers don't always follow the expected message order.
Always handle optional messages gracefully.


When connecting to a peer, the BitTorrent protocol expects the peer to send a Bitfield message first — telling you which pieces they have. 
But when I tested against a real peer, it sent an Unchoke message (ID=1) instead of a Bitfield (ID=5) and my client crashed. 
Turns out some peers skip the Bitfield entirely if they have all pieces and go straight to Unchoke — implying "I have everything, just ask me." 
The fix was to detect this case and create a full bitfield with all bits set to 1 (0xff), assuming the peer is a full seeder.
